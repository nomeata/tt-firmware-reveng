# The `.upd` tail and the `A:` (SYSTEM) partition — where A:'s content comes from

Research question: *what is the ~6.4 MB tail of `update3202MT.upd` (0x47ECCC..EOF), and
how does the pen's `A:` partition (`A:/SYSTEM`, `A:/VOIMG`, `A:/Language`, `*.lst`) get
populated?*

**Answer (short):** the tail is the **`to_udisk` file payload** of the `.upd` container —
7 named files (one system-voice archive + six language prompt WAVs) that the pen's own
updater writes onto `A:` after **reformatting A:** on every firmware update. Everything
else on `A:` (`SYSTEM/profile.dat`, `oidfilelist.lst`, `musiclist.lst`, `voicelist.lst`,
the FAT itself) is **generated by the firmware at runtime**. So yes — A:'s authentic
factory content is in the `.upd` and trivially extractable; and no — none of it is
strictly required for boot/discovery to work (see §5).

Conventions: `.upd` = the `update3202MT.upd` container (11,303,276 = 0xAC796C bytes). MT PROG
addresses = unified base 0x08009000, decomp in `out/decomp_named/`. Claims tagged
**Proven** (bytes/decomp cited) or **Inferred**.

---

## 1. Complete `.upd` container map (all sections, not just the 4 extracted)

Header layout (0xA4 bytes, checksummed; matches `fw/2N-Update3202/data/Anyka_UPD_2015-03-01.xml`).
All values below read from the file (**Proven**):

| hdr off | field | value | points to |
|---|---|---|---|
| +0x00 | magic | `ANYKA106` | |
| +0x08/+0x0c | partitions count/adr | 2 @ 0xA4 | `A:` UNSTANDARD 30 MB, `B:` STANDARD (0x18-B records) |
| +0x10/+0x14 | flash_ic count/adr | 125 @ 0x200 | NAND geometry table (0x40-B records) |
| +0x18/+0x1c | producer adr/size | 0x3400, 0x1CAFC | `producer.bin` (factory/recovery userware) |
| +0x20/+0x24 | NANDboot adr/size | 0x20000, 0x7E80 | `nandboot.bin` (SPL) |
| +0x28/+0x2c | **to_udisk dirs** count/adr | **2 @ 0x2200** | dir-mapping records, 0x210 B each (§2) |

| +0x88 | fs_reserved_blocks | 0x40 | |

| +0x90 | flag_updateself | 1 | |
| +0x94 | flag_Snowbirds_E | 1 | |
| +0x98 | hardwaremode | 0 (NANDFLASH) | |
| +0x9c | unknown | 0 | |

| EOF−0x40 | superid trailer | `ANYKA_ID` + `RAV_N0038` | version trailer, checked by the updater (§4.1) |

**Checksum algorithm (Proven, verified):** `upd_checksum_xor@0x0810543c` —
`sum += buf[i] ^ buf[i+2]` for `i = 0, 2, 4, … < len−2`. Verified: header (len 0xA0)
→ 0x739 ✓; every dir record (len 0x20C) and file record (len 0x10C) carries the same
checksum over itself as its last u32 — all 9 verified ✓.

## 2. The ~6.4 MB tail = the `to_udisk` payload (Q1)

The tail (0x47ECCC..0xAC796C) is **not** a FAT image, not a partition image, and not
unnamed raw audio: it is 7 individual files, each **named in the TOC at 0x2800** with
size + absolute `.upd` offset + checksum. Data areas are 0x200-aligned; all gaps
(0x47ECCC→0x47EE00, inter-file, and after the last file up to the `ANYKA_ID`/`RAV_N0038`
trailer) are zero padding. The tail is thereby **100% accounted for**. **Proven** (parsed):

Dir-mapping records @0x2200 (`{u32 bcompare; char pcpath[260]; char udiskpath[260]; u32 checksum}`):

| pcpath | → udiskpath |
|---|---|
| `voice` | **`A:VOIMG`** |
| `Language` | **`A:Language`** |

File records @0x2800 (`{char path[260]; u32 size; u32 adr; u32 checksum}`):

| # | path in `.upd` | → on pen | size | .upd offset | checksum |
|---|---|---|---|---|---|
| 0 | `voice\Chomp_Voice.bin` | `A:/VOIMG/Chomp_Voice.bin` | 0x41EBE4 (4,320,228) | 0x47EE00 | 0x39F |
| 1 | `Language\BatLowUpdateDUTCH.wav` | `A:/Language/…` | 0x98224 | 0x89DA00 | 0x391 |
| 2 | `Language\BatLowUpdateFRENCH.wav` | `A:/Language/…` | 0x2DE52 | 0x935E00 | 0x391 |
| 3 | `Language\BatLowUpdateGERMAN.wav` | `A:/Language/…` | 0x36124 | 0x963E00 | 0x30A |
| 4 | `Language\UpdateDUTCH.wav` | `A:/Language/…` | 0x6FBA0 | 0x99A000 | 0x413 |
| 5 | `Language\UpdateFRENCH.wav` | `A:/Language/…` | 0x45E86 | 0xA09C00 | 0x386 |
| 6 | `Language\UpdateGERMAN.wav` | `A:/Language/…` | 0x77D2C | 0xA4FC00 | 0x2A8 |

File formats (**Proven**, bytes + consumer decomp):

- **`Chomp_Voice.bin`** — the system-voice archive ("Chomp" = Anyka's MMI framework, cf.
  source path `chomp\update\s_set_firmware_updata.c` @0x080521dc). Layout: u32 table
  `word[0..48]` then concatenated **RIFF/WAVE files** (PCM mono 16 kHz 16-bit).
  `word[i]` = byte offset of voice *i*; `word[0] = 0xC4` doubles as table size (49·4)
  and start of voice 0; `word[48] = 0x41EBE4` = file size = end sentinel ⇒ **48 system
  voices** (ids 0..47). Consumer: `fwl_play_voice_by_id@0x080ab9ac` opens
  `A:/VOIMG/Chomp_Voice.bin` (string @0x080ab990), seeks `id·4`, reads `start`,`end`,
  plays `[start, end)`. This is every beep/jingle/system prompt the pen makes.
- **`Language/*.wav`** — plain RIFF/WAVE (with `bext` broadcast chunk): the spoken
  "update running" (`Update<LANG>.wav`) and "battery too low to update"
  (`BatLowUpdate<LANG>.wav`) prompts. Consumer: `lang_play_batlow@0x080523b4` builds
  `A:/Language` + (`/Update` | `/BatLowUpdate`) + `<LANGNAME>.wav` (strings @0x080524cc,
  0x080524d8, 0x080524e0); LANGNAME (e.g. `GERMAN`) comes from `B:/LanguageInfoMT.txt`
  parsed by `lang_get_num@0x08052508`.

## 3. Who writes A: — the update path, traced (Q2)

The pen updates **itself**: the PC (tiptoi Manager) merely drops the `.upd` onto the
USB-exposed `B:` drive. Full chain in MT PROG (**Proven**, all decomp in
`out/decomp_named/`):

1. **Detect** — `fw_update_cleanup@0x08052768`: scans `B:\` for files whose name matches
   a language entry's update-file name from `B:/LanguageInfoMT.txt` (default constant
   `Update3202MT.upd` @0x080ecf12); opens candidate, seeks `size−0x38`, requires trailer
   `RAV_` and compares the `N....` version digits against the running firmware's —
   `"Can update."` / `"Delete Update File."` @0x08052b10/0x08052b20. Requires battery
   ≥ 0x300 ADC, else plays `A:/Language/BatLowUpdate<LANG>.wav` and aborts. Deletes all
   other update files, stores `B:\<name>` in the update ctx (+0x28CC).
2. **Format A:** — `fw_update@0x08051fd4`: `fs_open(.upd)`, then logs
   **`"start format driver A"`** (@0x080520d8) and calls
   `fat_partition_create_format@0x08039d00` → `fat_format_volume@0x0803b310` (full FAT
   mkfs) on partition A. **A: is wiped on every update.** (Chicken-and-egg note: the
   `Update<LANG>.wav` announcement is played from the *old* A: before the format.)
3. **Unpack** — `fw_update_2@0x080f2838` (log prefix `PR_S`), in order:
   - `upd_unpack_headinfo@0x08105478`: read first 0xA4 bytes, verify `ANYKA106` magic +
     header checksum (`upd_checksum_xor` over 0xA0 == u32@0xA0).
   - `pr_p_read_partition_info@0x08105344` + `pr_b_recover_bin_info`: reload the NAND
     metadata blocks 61/62/63 (see nand-init-from-producer.md).
   - `pr_u_file_count@0x0810573c`: iterate the **to_NAND** records (hdr+0x38/+0x3c),
     compare/re-burn `PROG` and `codepage` as RB/UB block pairs.
   - **`upd_unpack_udisk_files@0x080f2760`**: iterate the **to_udisk file** records
     (hdr+0x30 count, +0x34 adr, stride 0x110). Per file:
     - `upd_unpack_udisk_dirs@0x08105524`: read the 0x110 record; take the path
       component before `\`; scan the dir records (hdr+0x28/+0x2c, stride 0x210)
       matching `pcpath`; target = `udiskpath` + remainder
       (`voice\Chomp_Voice.bin` → `A:VOIMG\Chomp_Voice.bin`).
     - `part_udisk_write_verify@0x080f2444`: mkdir every path component, create the
       file (`"PR_S udisk file path:%s"` @0x080f26d8), seek the `.upd` handle to the
       record's `adr`, stream in 0x1000-byte chunks, then read back and compare
       (`"PR_S udisk file compare success"`).
   - `upd_unpack_bootfile@0x081058a4`: rewrite the SPL + metadata blocks 59..63
     (`pr_br_write_boot`, `pr_p_write_maps`).

So **the `.upd` is the authoritative source of A:'s persistent content**, and the pen's
own firmware is the writer (the same is done at the factory by the PC production tool
via `producer.bin`'s PR_I host-streamed-FAT-image path or file writes — the burn tool
reads the same to_udisk section; see nand-init-from-producer.md §2.5. **Inferred** for
the factory tooling detail, Proven that producer offers the mechanism).

### What the firmware generates itself (not in the `.upd`)

- **The FAT on A:** — mkfs'd by the updater (§3.2) and *also* re-created on any boot
  where the partition scan fails: `fs_storage_mount_init@0x0803a484` →
  `fat_partition_create_format@0x08039d00`. **Proven.**
- **`A:/SYSTEM/profile.dat`** — created on demand by the profile writer
  `FUN_0804b8cc@0x0804b8cc`: `fs_open("A:/SYSTEM/profile.dat")` (UTF-16 string
  @0x080affd4); if missing or size ≠ 0x21B0, it mkdirs `A:/SYSTEM/` (UTF-16 string
  @0x0803c9f0, via `FUN_080ad6d8`), creates the file and writes a 0x21B0-byte default
  profile built by `FUN_080391e4@0x080391e4` (defaults include a UTF-16 `B:/` scan
  path at profile+0xD4C). Reader twin: `FUN_0804bb20`. **Proven.** So `A:/SYSTEM` is
  pure runtime state (settings/persistence), never shipped.
- **`a:/oidfilelist.lst`, `a:/musiclist.lst`, `a:/voicelist.lst`** — the discovery
  indexes, (re)written by the GME/media discovery scan
  (`udisk_gme_discovery@0x080f013c`; path constants selected in `FUN_080ae2c0`,
  strings @0x080ae394/0x080ae3a8/0x080ae3bc). **Proven** (confirmed dynamically).
- `A:/Firmware log file.bin` — runtime log (observed in a standby trace).

## 4. How A: gets populated (Q3)

**Geometry** (from the `.upd` partition record @0xA4, **Proven**): `A:` = UNSTANDARD,
**30 MB**, 512-B sectors ⇒ FAT16 volume, formatted by the firmware's own
`fat_format_volume`.

**Authentic factory-fresh content** = exactly the 7 extracted files:

```
A:/VOIMG/Chomp_Voice.bin
A:/Language/BatLowUpdateDUTCH.wav   A:/Language/UpdateDUTCH.wav
A:/Language/BatLowUpdateFRENCH.wav  A:/Language/UpdateFRENCH.wav
A:/Language/BatLowUpdateGERMAN.wav  A:/Language/UpdateGERMAN.wav
```

plus whatever the firmware then creates itself (`SYSTEM/profile.dat`, `*.lst`, log).
Extraction one-liner (offsets from §2):

```python
import struct
d = open('update3202MT.upd','rb').read()
for i in range(7):
    o = 0x2800 + i*0x110
    path = d[o:o+0x104].split(b'\0')[0].decode()
    size, adr = struct.unpack_from('<II', d, o+0x104)
    # target path: apply dir map voice->VOIMG, Language->Language; payload = d[adr:adr+size]
```

**Is it required?** **No** (Proven, by the firmware's own fallbacks):

- A: FAT missing/corrupt → `fs_storage_mount_init` formats it in place.
- `Chomp_Voice.bin` missing → `fwl_play_voice_by_id` returns silently on `fs_open == −1`
  (no system voices, no crash).
- `Language/*.wav` missing → only the update/bat-low spoken prompts fail
  (`lang_play_batlow` logs `[NO FILE ]`, returns false).
- `profile.dat` / `*.lst` are auto-created anyway.

So for GME discovery and normal operation an **empty (or even unformatted) A: is
sufficient**; the factory content only adds the audible system feedback. A byte-faithful
A: is simply the 7 extracted files (under `VOIMG/` and `Language/`) placed into a 30 MB
FAT16 superfloppy; everything else the firmware creates itself.
